# FinVest Platform - 研发计划（方案 C：创始人 + AI 模式）

## 总体规划

- **项目周期**: 约 6-8 个月（4个阶段）
- **开发模式**: 创始人独立 + Claude Code AI 全栈开发
- **核心策略**: MVP 优先，先跑通核心链路再逐步完善
- **基础设施**: Docker Compose 单机部署，不上 K8s
- **AI 推理**: 全部走外部 API（DeepSeek 为主），不自建 GPU
- **数据源**: AKShare 免费方案为主
- **Sprint 节奏**: 1 周一个 Sprint（一人模式迭代更快）

## 技术决策（方案 C 适配）

| 决策点 | 选择 | 原因 |
| --- | --- | --- |
| 消息队列 | Redis Stream（替代 Kafka） | 初期数据量小，减少运维复杂度 |
| AI 推理 | DeepSeek API 为主 | 性价比最高，统一网关支持随时切换 |
| ML 训练 | Google Colab + AutoDL | 免费/低价 GPU，无需自采 |
| 交易网关 | Python（替代 Go） | 减少技术栈，初期并发要求不高 |
| 部署方式 | Docker Compose 单机 | 用户量 < 1000 时足够 |
| 监控 | Prometheus + Grafana + Loki + Sentry | 全开源零成本 |
| 数据源 | AKShare 为主 | 免费覆盖全品类 |

## 技术风险与应对

| 风险 | 影响 | 应对方案 |
| --- | --- | --- |
| 创始人精力瓶颈 | 进度放缓 | 严格 MVP 优先，砍掉非必要功能 |
| 券商 API 接入受限 | 实盘功能延后 | 优先 Alpaca（美股，最开放），模拟盘先行 |
| AI 模型预测准确率不足 | 用户信任度下降 | 保守策略为默认，强调风险提示，持续迭代 |
| 行情数据延迟/中断 | 数据不可用 | AKShare + yfinance 多源备份，自动切换 |
| 单机性能瓶颈 | 用户量增长后响应慢 | 提前设计无状态架构，用户量 > 1000 时迁移 K8s |
| LLM API 成本超预期 | 运营成本上升 | 成本追踪仪表盘，及时切换更低价模型 |
| 缺乏代码 Review | 代码质量风险 | AI 自查 + 自动化测试 + 关键模块请朋友审查 |
